## VERIFIED facts + exact citations

1. **The omnibus linker = "libomni".** Buck's Omnibus javadoc (https://buck.build/javadoc/com/facebook/buck/cxx/Omnibus.html), verified quote:
   > "only links native nodes which have an explicit edge from non-native code as separate, and statically linking all other native nodes into a single giant shared library. This reduces the number of shared libraries considerably and also allows the linker to throw away a lot of unused object files."
   So: Python's many native extensions would otherwise produce a huge number of `.so`s; omnibus statically merges most native code into ONE giant `libomnibus.so`, keeping only the direct extension roots as separate `.so`s.

2. **It's a configurable `native_link_strategy`:** `merged` (uses omnibus) vs `separate` (pulls in shared libraries linked in the context of the rules that own them). Omnibus is described as experimental / for "absolutely huge C++ codebases," "normally can be safely turned off." (buck2 python toolchain docs; https://buck2.build/docs/prelude/rules/python/prebuilt_python_library/ and the .buckconfig docs https://buck.build/files-and-dirs/buckconfig.html)

3. **PUBLIC failure evidence (the citation that earns its place):** facebook/buck2 issue #62, "libomnibus changing weak symbol to undefined?" (https://github.com/facebook/buck2/issues/62) — VERIFIED real. Importing a `cxx_python_extension` fails with `ImportError: undefined symbol: _ITM_registerTMCloneTable` from `libomnibus.so`; `nm` shows libomnibus flipping that symbol to undefined (`U`) while `libc10.so`, `libtorch_cpu.so`, `libtorch_python.so` keep it weak (`w`). i.e. omnibus + PyTorch + a symbol-handling problem, on Meta's own tracker. Closed "not planned." Good evidence that native-merge symbol issues in Python+torch binaries are a documented, open thing.

4. **Link groups as the successor / more granular tool:** `link_group_map` in the buck2 prelude carves the linked native graph into multiple shared libraries (rather than one omnibus blob or fully separate). Matches Deep's recollection that libomnibus had problems → they moved to link groups. (Public API exists; semantics lightly documented — per the earlier prior-art pass, the in-source docs are thin.)

5. **Sibling precedent — Android native library merging (Meta eng blog, 2018):** https://engineering.fb.com/2018/01/23/android/android-native-library-merging/ and the demo repo https://github.com/fbsamples/android-native-library-merging-demo — same motivation (too many native `.so`s → merge to reduce count/size) and, per search summaries, the same hazard stated as: merging "works great as long as there are no common symbols between the libraries being merged."

## How to use it in the article (Fable)

- **§1 setting upgrade:** where the post currently says binaries are "carved into link groups" past the 2 GiB barrier, enrich it: Python training binaries pull in a large number of native shared libraries; Buck's omnibus (`merged` native_link_strategy) statically links most of that native code into a single giant `libomnibus.so` to keep the `.so` count and startup sane, leaving direct extension roots separate. When that merge can't be clean — prebuilt/unrelinkable extensions, common symbols, or the sheer 2 GiB size — you fall back to separate copies or carve the graph into link groups. That is precisely the world in which the same native library (here the RDMA verbs stack) can end up present in two places in one binary. Cite the omnibus javadoc; optionally nod to buck2#62 as public evidence that omnibus symbol handling in torch binaries is a known sharp edge. Keep it SETTING — do not claim omnibus was the incident's mechanism.
- **§7 the two-ecosystems symmetry (strong):** the monorepo manages Python's native-`.so` explosion by MERGING (omnibus/link groups) — and hits split-state when the merge has common symbols. The PyPI wheel ecosystem manages the same explosion by VENDORING a private copy per wheel (auditwheel) — and hits Route B when those copies co-load. Two ecosystems, the same underlying pressure (Python brings too many native libraries), opposite mitigations, and the same disease (two live copies of one library's state) leaking out the side of each. This makes §7 land even harder.
